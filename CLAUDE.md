## ⛔ 제1원칙 — 하드웨어를 안전하게 쓰게 할 책임은 이 소프트웨어에 있다

이 데몬은 **실물 하드웨어**(SY-01B 시린지 펌프·회전 밸브·솔레노이드 밸브)를 직접 구동한다.
펌프는 액을 빨고 뱉고, 플런저가 물리적으로 움직이며, 걸리면(스톨) 모터가 과부하된다.
**하드웨어가 "잘못" 반응하는 건 없다 — 하드웨어를 안전하고 견고하게 다룰 책임은 전적으로 이 소프트웨어에 있다.**
운영자·하드웨어를 탓하지 말고, 소프트웨어가 모든 물리 상태를 안전하게 흡수하도록 설계한다.

이 원칙은 새 기능·수정·리뷰 때마다 먼저 떠올린다. 구체 규칙:

### 1. 무한/과도 대기 금지 — 모든 하드웨어 대기는 상한이 있고, 멈춤을 즉시 감지한다
- 펌프가 "멈췄다(ready 비트 set)"고 신호하면 **즉시** 알아채고 판정한다. 에러코드가 붙어 있다고
  "아직 안 끝났다"로 오해해 계속 폴링하면 안 된다 (2026-07-20 사고: 플런저가 9583에서 멈췄는데
  Code 15를 busy로만 봐 33초 매달리다 transient 실패 → 배출조차 못 감).
- 모든 모션은 `motion_timeout` 상한을 갖는다(불변식 F1). 상한은 "정상 완주 시간"이지 "행 감지 시간"이
  아니다 — **정지를 먼저 감지**하고, 그게 안 될 때의 최후 방어선이 timeout이다.

### 2. 거짓 성공 금지 (EP-03) — 실제로 안 된 걸 됐다고 보고하지 않는다
- 버퍼가 비었다고, 에코가 왔다고, 예외를 삼켰다고 성공으로 보지 않는다. **실제 상태 프레임(`/0…`)을
  parse_status 로 확인**해야 성공이다.
- 스톨/오버로드를 조용히 성공으로 위장하면(예: 20%만 토출됐는데 완료 보고) 더 큰 사고다(안 나온 음료를
  "나왔다"고 보고). 정직한 실패가 거짓 성공보다 항상 낫다.

### 3. 근접 완료는 완료로, 큰 미달은 안전 복구 — 두 경우를 구분한다
- 물리적으로 **사실상 완료된** 동작(목표의 충분히 높은 %까지 도달)은 완료로 인정하고 진행한다
  — 마지막 몇 스텝에 매달려 실패로 던지지 않는다.
- **크게 미달**한 채 멈췄으면 진짜 문제(막힘·빈 통·기계 한계)다 → 안전하게 정지 → 원점 복귀(재초기화)
  → 명확히 실패 보고. **성공으로 위장 금지.**
- 판정 기준(허용 %·복구 절차)은 하드코딩 추측이 아니라 **실측(프로브)으로 확정**한다.

### 4. 절대 이동 선호 — 위치 드리프트에 안전하게
- 플런저 이동은 상대(`P`/`D`)보다 **절대(`A{steps}`/`A0`)**를 선호한다(v1.1.0 방식). 상대 이동은
  시작 위치가 어긋나면 목표를 벗어나고, 재시도 때 위치가 누적돼 물리 한계를 넘는다(2026-07-20).
  절대 이동은 현 위치와 무관하게 목표가 확정된다.

### 5. 스톨/오버로드 복구 — 재시도 전에 원점을 되찾는다
- 오버로드(Code 11·9 등) 후 무작정 같은 명령을 재전송하지 않는다. 먼저 안전 정지(TR)·재초기화로
  **홈 기준을 되찾은 뒤** 절대 이동으로 재시도하거나 정직하게 실패한다.

### 6. 하드웨어 거동은 추측하지 말고 프로브로 확인한 뒤 다룬다
- SY-01B 상태바이트·에러코드·스톨/복구 거동은 매뉴얼 + **실기기 프로브 실측**으로 확정한다
  (예: 일회성 `scripts/pump_motion_probe.py`(모션)·`scripts/pump_link_diag.py`(읽기전용 링크) 프로브로
  상태바이트·위치·ready 비트를 캡처). 추측으로 폴링·판정
  로직을 짜는 것 자체가 "하드웨어를 안전하게 다루지 못하는" 것이다.

### 7. 펌웨어 기본값 의존 금지 — 물리 결과를 바꾸는 파라미터는 항상 명시한다
- 파라미터를 생략한 명령은 "아무 일도 지정 안 함"이 아니라 **펌웨어 기본값을 지정한 것**이다.
  기본값이 우리 배관과 맞는지는 펌웨어가 모른다 (2026-07-21 사고: `Z{힘}R` 포트 생략 →
  펌웨어 기본 "흡입=포트1·배출=마지막 포트"로 돌아 **매 초기화마다 포트1 향료를 빨았다 폐기**
  + `I12R`(대기 개방 포트) 주차로 상시 누액 — QA "초기화시 흡입/배출 포트 변경").
- 새 명령을 보낼 땐 매뉴얼에서 **생략된 파라미터의 기본값이 무엇인지 반드시 확인**하고, 물리
  결과(어느 포트에서 빨고 어디로 뱉나·힘·속도)를 바꾸는 파라미터는 **전부 명시**한다.
- 값의 출처는 하드코딩이 아니라 **서버 해석값(설정 파생)** — 포트 배치처럼 배관·모드에 따라
  달라지는 값은 서버(포트 매핑 SoT)가 스텝에 실어 보내고, pi 는 받은 값을 쓴다(부재=하위호환).
- **밸브가 쉴 땐 언제나 배출구(output)를 보게 한다** — 대기 개방 포트(air) 주차는 잔여액
  배수 경로다(2026-07-21 확정 규칙).
